# 映射链路产品级审查方法

日期：2026-03-23

## 1. 目的

这份文档用于约束后续对“映射补录 / 映射规则治理 / 映射回刷 / 后处理生效”整条链路的排查方法，避免继续采用“发现一个可疑点就单点追查”的命中式方式。

后续排查以“成熟产品”的标准进行，不只判断代码是否能运行，而要判断：

- 用户是否存在明确、可闭环、可恢复的操作路径
- 系统是否对每一种状态给出一致、可解释的语义
- 边界条件下是否会产生静默截断、假成功、误导性默认值、不可见失败或不可修复状态
- 前端、API、服务层、存储层、后处理层之间是否保持同一业务语义

## 2. 审查对象

本次只审查与“映射”直接相关的产品链路，不扩展到下载、打包、浏览器安装等无关模块。

### 2.1 前端对象

- `desktop_app/renderer.js`
- `desktop_app/renderer/mappings.mjs`
- `desktop_app/renderer/mappings.test.js`
- 待补映射列表、规则录入区、规则表、批量导入草稿区、冲突确认弹窗

### 2.2 API / 服务层对象

- `desktop_backend/app_backend.py`
- `desktop_backend/app_service.py`

关注接口：

- `GET /api/mappings`
- `POST /api/mappings/preview`
- `POST /api/mappings`
- `POST /api/mappings/reprocess-pending`
- `POST /api/records/:id/reprocess`

### 2.3 存储层对象

- `peap/streaming_store.py`

关注表与投影：

- `mapping_entries`
- `mapping_pending`
- `records`
- `record_revisions`
- `jobs`
- `job_events`

### 2.4 后处理与回刷对象

- `peap/streaming_postprocess.py`
- `peap/streaming_ingest.py`

关注：

- 规则匹配
- 规则优先级
- 歧义处理
- 循环引用
- 状态分类
- 回刷后是否真正消除 `pending_mapping`

## 3. 审查原则

### 3.1 以操作为中心，不以函数为中心

排查单位必须是“真实用户可能发起的一次操作”，而不是“某个函数看起来可疑”。

一个操作至少要覆盖：

1. 用户如何进入这个操作
2. 前端默认填了什么
3. 前端提交了什么 payload
4. API 返回什么状态码与响应体
5. 服务层创建了什么任务/没有创建什么任务
6. 存储层如何变更
7. 后处理是否真的生效
8. 最终 UI 是否能让用户理解发生了什么

### 3.2 以产品闭环为标准

只要某种真实数据状态进入系统后，用户没有一条可行路径把它从错误/待补状态修复到正确状态，就视为产品级缺陷。

### 3.3 不接受静默截断

任何硬编码上限、分页、去重、过滤、状态排除，只要用户无法感知，就按产品缺陷审查。

### 3.4 不接受假成功

任何“接口返回成功 / UI提示成功 / 实际没有完成业务动作”的情况，优先级按严重缺陷处理。

### 3.5 不接受语义分裂

同一业务概念在不同层必须一致。例如：

- `pending_mapping` 到底表示“缺类型”还是“任意映射问题”
- `conflict` 到底是否参与回刷
- `group_type` 到底是否是一级修复路径还是回退路径

前端、后端、存储、后处理不能各说各话。

## 4. 业务对象与状态模型

后续所有检查都围绕这四类业务对象展开：

### 4.1 记录

核心状态：

- `ready`
- `pending_mapping`
- `conflict`
- `skipped`
- `parse_failed`
- `postprocess_failed`

### 4.2 映射规则

四类规则：

- `transferor -> group_name`
- `transferor -> source_type`
- `group -> group_name`
- `group -> source_type`

### 4.3 待补项

`mapping_pending` 是“记录的当前版本仍无法进入导出”的产品投影，而不是独立业务实体。排查时必须确认：

- 它何时产生
- 何时 resolve
- 是否可能重复
- 是否可能漏掉
- 是否可能残留历史脏项

### 4.4 回刷任务

`mapping_refresh` 是规则治理的执行器，不只是后台线程。排查时必须确认：

- 创建条件
- 覆盖范围
- 并发锁
- 失败可见性
- 对 `ready/pending/conflict` 的一致性

## 5. 操作矩阵

后续检查必须按下列操作矩阵逐条执行，不允许跳项。

### A. 规则创建

1. 手工新建 `transferor -> group`
2. 手工新建 `transferor -> type`
3. 手工新建 `group -> group`
4. 手工新建 `group -> type`

每项都要检查：

- 默认表单值是否合理
- payload 是否正确
- preview 是否与 save 语义一致
- 保存后是否影响正确集合

### B. 规则更新与覆盖

5. 保存完全相同规则
6. 保存同源同维度同目标字段但改目标值
7. 保存同源不同 `target_field`
8. 保存同源不同 `match_field`
9. 覆盖确认取消
10. 覆盖确认通过

每项都要检查：

- preview 的 `mode`
- `conflict` 判断
- 覆盖提示文案
- save 后实际入库结果
- refresh 影响范围是否与 preview 一致

### C. 待补入口

11. 从单条待补记录导入规则
12. 批量导入待补记录到草稿区
13. 草稿区规则类型切换
14. 草稿区批量保存
15. 草稿区存在重复规则
16. 草稿区部分成功、部分失败

每项都要检查：

- 默认规则类型是否符合最常见待补原因
- `sourceName` 是否总能被正确推导
- 去重是否误吞规则
- 失败反馈是否能定位到具体草稿

### D. 记录重处理

17. 单条记录重处理
18. 批量待补重处理
19. 保存规则后自动回刷
20. 回刷期间再次保存规则

每项都要检查：

- 锁是否正确
- 是否存在假成功
- 任务是否可追踪
- 结果状态是否正确落到 `ready/pending/conflict`

### E. 数据形态边界

21. 只有公司名，没有集团
22. 只有集团，没有公司名
23. 公司名和集团都存在
24. 公司名命中多条规则
25. 集团链存在循环
26. 集团链存在分叉歧义
27. 规则命中 `ready` 记录
28. 规则命中 `pending_mapping` 记录
29. 规则命中 `conflict` 记录
30. 命中数量超过系统限制

每项都要检查：

- 是否存在可修复路径
- 是否存在静默排除
- 是否存在 UI 不可见失败

### F. 规模与恢复

31. 待补项超过前端可见上限
32. 受影响记录超过回刷上限
33. 后端重启时存在运行中的 `mapping_refresh`
34. 数据库文件丢失后的映射页恢复
35. 历史脏规则与 legacy 数据兼容

## 6. 每项操作的判定标准

每个操作都必须按以下六个维度判定：

### 6.1 可达性

用户能否从现有 UI 自然到达这条操作路径，而不依赖内部脚本或手工改数据库。

### 6.2 正确性

系统最终结果是否符合规则语义。

### 6.3 一致性

前端文案、API 返回、任务状态、记录状态、后处理结果是否一致。

### 6.4 可解释性

失败时，用户能否知道：

- 为什么失败
- 哪条规则失败
- 是否已经保存
- 是否已经回刷
- 还有哪些记录没处理

### 6.5 可恢复性

用户能否不借助工程师，自己走下一步把状态修正回来。

### 6.6 可扩展性

数据量变大后，当前行为是否仍然保持真实语义，而不是靠静默截断“装作完成”。

## 7. 证据采集方式

每个结论必须落在以下证据之一，不能只靠阅读印象：

### 7.1 真实复现

优先级最高。包括：

- 真实 `AppService` 调用
- 真实 HTTP handler 调用
- 真实后处理函数调用

### 7.2 自动化测试现状

要明确：

- 当前已有测试覆盖了什么
- 哪个边界没有被测试覆盖

### 7.3 代码定位

每个结论至少要落到：

- 触发入口
- 核心判断
- 结果落地

三类位置之一。

### 7.4 不足以成立的证据

以下不单独作为结论依据：

- “看起来像有问题”
- “大概率会这样”
- “从别处类似 bug 推断”

这些只能作为下一轮复现的假设，不能直接写成问题结论。

## 8. 输出格式

后续每发现一个问题，都按以下模板记录：

1. 操作名称
2. 复现步骤
3. 预期的成熟产品行为
4. 实际行为
5. 根因位置
6. 影响范围
7. 严重度

## 9. 本轮后续执行顺序

后续实际排查严格按以下顺序，不允许跳到某个偶然看到的点：

1. 规则创建四种类型
2. 规则更新与覆盖
3. 待补入口单条/批量
4. 单条重处理与批量回刷
5. 公司/集团不同数据形态
6. 歧义与循环
7. `conflict` 与大规模数据边界
8. 恢复与重启场景

## 10. 当前约束

在这份方法文档之后，后续排查必须满足两条约束：

- 不再“看到哪里可疑就追哪里”，必须先定位它属于操作矩阵的哪一项
- 没有真实复现或明确代码证据，不把猜测写进正式报告
