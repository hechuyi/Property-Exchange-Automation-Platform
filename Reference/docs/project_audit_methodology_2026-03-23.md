# 项目级产品审查方法

日期：2026-03-23

## 1. 目的

这份文档用于约束后续对整个项目的审查方法。目标不是继续按“看到哪里可疑就查哪里”的方式做代码巡检，而是按成熟产品的标准，把真实用户可能发生的每一种操作逐条拆开，逐条验证，并要求每个结论都落到可复现证据上。

这里的“成熟产品标准”指的不是“代码能跑”，而是：

- 用户有明确的主路径、边界路径和恢复路径
- 前端、API、服务层、存储层、后台任务、文件系统和打包产物对同一业务语义保持一致
- 失败不会被伪装成成功
- 上限、截断、降级和回退不会静默发生
- 数据规模增大时，行为仍然可解释、可追踪、可恢复

## 2. 产品边界

本项目当前的产品边界以仓库现状为准：

- 产品主入口：`desktop_app/` + `desktop_backend/`
- 运行时核心引擎：`peap/`、`peap_core/`、`peap_parsers/`、`peap_postprocess/`
- 维护脚本与打包脚本：`scripts/`
- 文档与设计说明：`docs/`

旧 CLI 壳层不是主产品入口，但它们调用的引擎代码仍在桌面产品热路径上，因此不能忽略其运行时行为。

## 3. 审查原则

### 3.1 以用户操作为单位

排查单位必须是“用户可能发起的一次真实操作”，而不是某个函数或文件。

每个操作都必须沿着以下链路检查：

1. 用户如何进入
2. 前端默认值与可见状态
3. API 请求 payload
4. API 返回码与返回体
5. 服务层逻辑与任务创建
6. 存储层状态变化
7. 文件系统副作用
8. 后台任务和事件投影
9. UI 最终反馈

### 3.2 以闭环为标准

只要某一类真实数据状态进入系统后，用户没有一条可执行、可理解、可恢复的路径将其修正到正确状态，就按产品缺陷处理。

### 3.3 不接受静默行为

以下行为一律按缺陷审查：

- 静默截断
- 静默降级
- 静默回退
- 静默忽略输入
- 静默排除部分数据

### 3.4 不接受假成功

任何“接口成功 / UI 提示成功 / 实际没有完成关键动作”的情况，优先级按严重问题处理。

### 3.5 不接受语义分裂

同一业务概念在不同层必须一致。例如：

- `ready` 是否一定可导出
- `pending_mapping` 是否等于“缺类型”
- `conflict` 是否仍属于可重处理记录
- “安装完成”是否等于“下载动作已可用”
- “任务开始”是否等于“任务账本中已有任务对象”

### 3.6 语义不变量筛查总纲（冻结版）

以下口径仅作摘要，完整冻结定义以 [语义不变量系统性筛查设计](/Users/rtoc/Documents/WorkSpace/Property-Exchange-Automation-Platform/docs/superpowers/specs/2026-03-25-semantic-invariant-screening-design.md) 为 canonical source。

后续审查在方法论层统一收敛到三条主线与六类不变量：

- 任务语义主线：任务生命周期不变量
- 范围语义主线：范围一致性不变量
- 对象身份主线：对象身份不变量
- 横切项：读写副作用不变量、默认值 / 回退路径不变量、截断 / 上限 / 容量不变量

问题归类遵守“只在三条件同时满足时新开 5.x”：既有报告没有明确写过、能够稳定复现、且问题本质已经脱离既有主线裂缝。否则统一并入既有主线的边界量化，记成新的 `7.x` 或 `11.x`。

完成标准不是“扫描完所有 `5.x`”，而是三条 release-gate 条件同时成立：

1. 主链任务在所有终态下只剩一套一致解释。
2. 用户看到的集合、操作作用的集合、提示解释的集合一致。
3. 失败对象在读、修、重试、重处理四种路径下身份稳定且可追溯。

## 6. 每个操作的判定标准

每项操作必须按以下七个维度判定：

### 6.1 可达性

用户能否自然进入这条路径，而不是依赖工程师手工修改环境。

### 6.2 正确性

系统最终状态是否符合业务语义。

### 6.3 一致性

前端、API、任务、数据库、文件系统和最终展示是否一致。

### 6.4 可解释性

用户是否能知道发生了什么，失败时是否知道为什么失败。

### 6.5 可恢复性

用户是否可以自己继续下一步把状态修复回来。

### 6.6 可观测性

任务、日志、状态、事件是否足以让人追踪问题。

### 6.7 可扩展性

数据规模增大后，当前语义是否仍成立，而不是被上限或静默截断破坏。

## 7. 证据标准

每个问题结论至少要满足以下证据之一：

### 7.1 真实复现

优先级最高，包括：

- 真实 `AppService` 调用
- 真实 HTTP handler 调用
- 真实前端纯逻辑调用
- 真实后处理函数调用

### 7.2 自动化测试现状

要明确指出：

- 已有测试覆盖了什么
- 哪个边界没有测试覆盖

### 7.3 明确代码定位

至少给出：

- 触发入口
- 核心判断
- 结果落地

中的一个或多个位置。

### 7.4 不可接受的“伪证据”

以下内容不能单独构成正式问题结论：

- 直觉判断
- 代码印象
- 未复现的推测
- “大概率如此”

这些只能作为下一轮复现假设。

## 8. 输出格式

后续每发现一个问题，都按同一模板记录：

1. 所属子域
2. 操作名称
3. 复现步骤
4. 成熟产品的预期行为
5. 实际行为
6. 根因位置
7. 影响范围
8. 严重度

1. 启动与首屏
2. 设置与工作区
3. 浏览器运行环境
4. 一键执行
5. 任务系统
6. 数据记录

## 10. 当前约束

从这份文档开始，后续排查必须遵守两条约束：

- 不再“逮到哪个查哪个”，必须先归类到某个子域和操作编号
- 没有复现或明确代码证据，不把猜测写入正式报告
