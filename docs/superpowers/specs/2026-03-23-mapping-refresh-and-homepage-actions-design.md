# 映射回刷治理与首页主操作重排设计

## 背景

当前桌面产品在映射治理和首页操作排布上有三个已暴露的问题。

第一，新增映射规则后的自动回刷虽然存在，但它的产品语义不够强。用户无法在保存前看到“这条规则是否会覆盖已有规则”“会影响多少条记录”，因此当前行为更像后台静默重跑，而不是一个可审计、可预期的规则治理流程。

第二，重处理能力粒度过细。现在只有单条记录重处理，挂在待补映射列表项里；当待补记录积累后，操作员缺少一个稳定的“批量把所有待补项再跑一遍”的入口。

第三，首页首屏的信息优先级不对。高频动作虽然都在首页，但 `一键执行 / 导出 / 手动导入` 被纵向堆叠，常见笔记本视口下导出和其他高频动作可能落到首屏以下，导致“关键操作需要先滚动才能找到”。

## 目标

1. 映射规则保存前提供冲突预警，并允许用户显式选择取消或覆盖。
2. 在映射页提供“当前所有待补映射记录”的一键批量重处理入口。
3. 首页首屏固定露出高频主动作，但批量重处理仍保留在映射页，不上首页。
4. 不新增独立“重处理页面”；批量重处理仍归属映射页这一侧边栏入口。

## 非目标

1. 不改变左侧导航的信息架构；仍保留“总览 / 任务 / 数据记录 / 映射补录 / 设置”。
2. 不把规则编辑器整体搬到首页。
3. 不引入新的前端框架。
4. 不重写映射数据模型，只在现有 `mapping_entries + mapping_refresh` 机制上补前置分析和批量入口。

## 设计决策

### 1. 映射规则保存改成“预检查 -> 冲突确认 -> 保存回刷”

当前 `upsert_mapping` 直接入库并启动回刷任务，缺少显式的保存前分析。改造后，保存流程拆成两步：

1. 前端先请求一个“映射预检查”接口。
2. 后端返回：
   - 是否存在同源规则冲突
   - 冲突规则的当前值
   - 这次保存预计影响的记录数
   - 这次保存是新增、更新还是覆盖
3. 若没有冲突，前端直接保存。
4. 若存在冲突，前端弹出确认层，提供：
   - `取消`
   - `覆盖旧规则并启动回刷`

这里的“冲突”定义为：相同 `match_field + source_name + target_field` 已存在，但目标值与本次提交不同。例如已有 `group -> source_type: 华润 -> 地方国企`，再次保存 `华润 -> 央企`，则必须触发警告。

### 2. 批量重处理入口放在映射页，不新增独立页面

批量重处理不做新页面，也不挂到首页主导航。它保留在 `映射补录` 面板中，位置放在待补映射区块顶部或其操作条中，按钮文案为：

- `一键重处理待补映射`

它的作用范围固定为“当前所有 `pending_mapping` 记录”，不与记录页筛选条件或映射页搜索条件耦合。这样有两个好处：

1. 语义稳定，用户不需要猜测“当前页筛选是否影响重处理范围”。
2. 后端接口可以直接基于规范化状态视图批量取数，不依赖前端当前界面状态。

执行后，系统启动一个 `mapping_refresh` 类型的后台任务，并返回：

- 总待处理记录数
- 实际启动重处理的记录数
- 任务 ID

### 3. 首页首屏只保留下载/导出/手动导入主动作

首页首屏的主动作区重新分层：

第一层是“当前处理状态”与任务进度。

第二层是固定首屏可见的三个主动作卡片：

1. `一键执行`
2. `导出 Excel`
3. `手动导入解析`

批量重处理不放首页，仍放在映射页。首页只需要把下载、导出、手动导入这三类日常操作整理到常见笔记本首屏内，并保留一个明显的“待补映射”入口把用户带到映射页。

### 4. 映射页的职责分层

映射页分成三块职责：

1. 待补映射列表
2. 批量重处理与单条重处理动作
3. 映射规则治理（新增/更新、冲突预警、已保存规则表）

其中：

- 单条 `仅重处理` 保留
- 新增一个批量 `一键重处理待补映射`
- 保存规则前先做预检查
- 已保存规则表继续保留筛选和搜索

这样映射页的操作逻辑是清晰的：待补项处理和规则治理在同一业务域内完成，但不会再把首页变成规则编辑台。

## 后端行为设计

### 1. 新增映射预检查接口

新增一个只读分析接口，例如：

- `POST /api/mappings/preview`

输入：

- `source_name`
- `match_field`
- `target_field`
- `target_value`

输出：

- `conflict`: 是否冲突
- `mode`: `create | update | overwrite`
- `existing_entry`: 若冲突，返回旧规则摘要
- `affected_count`: 预计受影响的最新记录数
- `affected_pending_count`: 其中当前仍为 `pending_mapping` 的数量

### 2. 新增批量重处理待补映射接口

新增一个批量入口，例如：

- `POST /api/mappings/reprocess-pending`

后端直接查询当前所有 `pending_mapping` 最新记录，创建一个 `mapping_refresh` 任务批量重处理。该任务不要求传入前端筛选条件。

### 3. 保持现有单条重处理接口

现有：

- `POST /api/records/{record_id}/reprocess`

继续保留，用于待补列表里的逐条操作。

## 前端行为设计

### 1. 映射规则保存交互

点击“保存规则”后：

1. 先调用预检查接口
2. 无冲突时直接提交保存
3. 有冲突时弹出确认层，显示：
   - 旧规则值
   - 新规则值
   - 预计影响记录数
4. 用户选择：
   - 取消
   - 覆盖并保存

### 2. 映射页批量重处理

在待补映射列表上方或同级操作区新增按钮：

- `一键重处理待补映射`

点击后直接启动批量任务，并在页面状态区显示：

- 已启动批量重处理任务
- 待处理总数
- 任务 ID

### 3. 首页首屏重排

首页首屏改成两段：

1. 顶部进度与状态卡
2. 三个并列主动作卡：
   - 一键执行
   - 导出 Excel
   - 手动导入解析

同时保留显著的待补映射入口，将用户带到映射页执行批量重处理或规则治理。

## 验证策略

1. 服务层测试锁定：映射预检查能识别新增、更新和冲突覆盖。
2. 服务层测试锁定：批量重处理待补映射只取当前 `pending_mapping` 最新记录。
3. 前端测试锁定：首页首屏含三个主动作卡片。
4. 前端测试锁定：映射规则冲突时显示确认流程而不是直接保存。
5. 回归验证：单条重处理、映射保存后自动回刷、任务页展示都保持正常。
