# 语义不变量系统性筛查设计

## 背景

`docs/real_operation_test_report_2026-03-23.md` 已经证明，当前产品的主要裂缝不是孤立页面 bug，也不是单个 API 缺字段，而是同一业务对象在 store、service、API、frontend 四层里被不同方式解释。继续按“页面 / 接口 / 目录”泛扫，只会不断重复看到同一条裂缝的不同表象，却不能稳定收口问题本质。

因此，后续筛查必须从“功能点巡检”切换到“语义不变量验证”。排查单位不再是“记录页”“导出页”“任务页”，而是“一个语义不变量 + 一个最小对象样本 + 一条最短数据流”。

## 目标

1. 把后续系统性筛查固定在三条总主线上：任务语义、范围语义、对象身份。
2. 让每一轮排查都能回答“同一对象在四层里是否仍是同一个东西”。
3. 用最小复现、代码因果和用户后果三类证据，区分新主线问题与既有主线边界。
4. 用可复跑的测试和文档更新，把筛查从一次性发现转成可重复验证。

## 非目标

1. 不再按目录做无边界泛扫。
2. 不先做大样本数据质量统计。
3. 不从文案或 UI 表象出发倒推问题。
4. 不把“缺测试”本身当成用户可感知问题。
5. 不在同一轮同时追两个相邻怀疑点。

## 设计决策

### 1. 六类不变量统一挂到三条总主线

后续筛查统一使用六类不变量组织证据：

- 任务生命周期不变量
- 范围一致性不变量
- 对象身份不变量
- 读写副作用不变量
- 默认值 / 回退路径不变量
- 截断 / 上限 / 容量不变量

但报告骨架只保留三条总主线：

- 任务语义主线：受理、创建、运行、完成、失败、中断、提示、事件、计数
- 范围语义主线：列表、筛选、分页、导出、空态、统计、总数、当前页、默认值
- 对象身份主线：原始文件、归档副本、失败对象、重处理对象、去重 token、恢复入口

六类不变量中，`任务生命周期` 直接挂到任务语义主线，`范围一致性` 直接挂到范围语义主线，`对象身份` 直接挂到对象身份主线；`读写副作用`、`默认值 / 回退路径`、`截断 / 上限 / 容量` 作为横切项附着在这三条主线上解释。

### 2. 固定筛查顺序，以发布可信度为准

每一轮系统性筛查按以下顺序推进：

1. 先查任务是否存在、何时终结、终结类型如何解释。
2. 再查用户看到的集合、操作真正作用的集合、空态 / 提示解释的集合是否一致。
3. 再查失败对象在读、修、重试、重处理四条路径下是否保持同一身份。
4. 只有前三层收束后，才继续映射治理闭环。
5. 纯前端同步或呈现类问题最后处理。

这个顺序来自发布口径，而不是修 bug 的便利性。只要前三层没有收束，产品仍属于“局部可用，但整体不可信”。

### 3. 每一轮只追一个不变量怀疑点

每轮检查必须显式写出一句真问题，例如：

“当前记录页视图、实际导出集合、空导出提示阻塞计数，是否来自同一过滤范围。”

然后只允许选择一个最小对象样本，例如：

- 单任务
- 单失败页
- 单条待补记录
- 一宽一窄两条业务类型记录

数据流优先走：

- `AppService`
- `StreamingStore`
- 当前前端纯逻辑

只有在 HTTP 返回码或 handler 路由本身属于语义的一部分时，才引入 `app_backend` 和真实 HTTP。只有在纯逻辑无法说明问题时，才做浏览器回放。

### 4. 问题归类遵守“新主线门槛”

只有同时满足以下三条，才新开 `5.x`：

1. 既有报告没有明确写过。
2. 能稳定复现，不依赖偶发时序。
3. 根因或问题本质已经脱离既有主线裂缝。

否则统一并入既有主线的边界量化，记成新的 `7.x` 或 `11.x`。编号不是成果，主线收口才是成果。

### 5. 证据标准固定为三段式

每条结论都必须能回答三件事：

1. 最小复现摘录：它稳定发生了吗。
2. 代码证据：为什么会这样，而不是偶发。
3. 用户后果：用户到底会误解什么、误操作什么、失去什么恢复能力。

不以“设计不优雅”“测试不充分”作为结论，不写无用户后果的工程感想。

## 冻结来源与引用规则

本文件是语义不变量筛查的 canonical frozen source。后续方法论文档和真实操作报告只允许摘要、引用和回链，不得各自重写一套不同表述的治理规则；如需说明判定口径，应回链到本文件，而不是在别处平行定义。

## 完成标准

系统性筛查不是“把 5.x 再扫一遍”，而是以下三件事同时成立：

1. 主链任务在所有终态下只剩一套一致解释。
2. 用户看到的集合、操作作用的集合、提示解释的集合一致。
3. 失败对象在读、修、重试、重处理四种路径下身份稳定且可追溯。

只要这三件没有同时成立，筛查就不能视为真正收口。

## 验证策略

1. 任务语义主线优先用真实 `AppService` 和 `StreamingStore` 复现，只有在 HTTP 码语义重要时才引入 `build_handler()`。
2. 范围语义主线优先用 `desktop_app/renderer/*.mjs` 的纯查询构造逻辑对照 `AppService.list_records()` 与 `run_export()`。
3. 对象身份主线优先比较 `records`、`job_events`、物理文件路径和 candidate token 的一致性。
4. 每一轮只产出一个结论：新开 `5.x`，或并入既有 `7.x / 11.x` 边界。

## 6. 冻结锚点

后续所有语义不变量筛查都按这组锚点归档：

- 任务语义主线
- 范围语义主线
- 对象身份主线
- 只在三条件同时满足时新开 5.x
- 完成标准

其中，“只在三条件同时满足时新开 5.x”指的是：既有报告没有明确写过、能够稳定复现、且问题本质已经脱离既有主线裂缝；“完成标准”指的是主链任务终态解释一致、集合语义一致、失败对象身份在读/修/重试/重处理四条路径下稳定可追溯。
